## Overview

ChoirOS Sandbox implements an **actor-based backend with REST API**, combining the actor model with event sourcing to build a structured, concurrent server application in Rust.

## Architecture

The crate is organized as a **module declaration hub** — `lib.rs` exposes nine public modules without re-exporting any concrete types, enforcing clear namespace boundaries:

- **`actors`** — Core actor definitions and message types. This is where individual actor structs, their mailboxes, and message-handling logic live.
- **`supervisor`** — Actor lifecycle management and supervision trees. Supervisors monitor child actors and implement restart/recovery strategies, following the classic "let it crash" philosophy.
- **`app_state`** — Shared application state, likely holding actor handles/addresses and any global resources needed across the system.
- **`api`** — REST API layer that bridges HTTP requests to actor messages, translating external calls into the internal message-passing protocol.

## Supporting Modules

- **`baml_client`** — Auto-generated client code (Clippy lints suppressed), likely providing typed interfaces to an LLM or external AI service via the BAML code generation tool.
- **`tools`** — Utility functions available to actors (likely tool-use capabilities for AI agents).
- **`markdown`** — Markdown processing, probably for formatting actor-produced content.
- **`observability`** — Telemetry, tracing, and logging infrastructure for monitoring actor behavior.
- **`runtime_env`** — Runtime configuration and environment variable management.

## Design Patterns

The system follows a **layered actor architecture**: external HTTP requests enter through `api`, are dispatched as messages to `actors`, which are managed by a `supervisor` hierarchy. Event sourcing provides durability and replayability of actor state transitions. The `app_state` module serves as the glue layer, holding actor addresses so the API layer can route messages to the correct actors.

The strict module-boundary design (no re-exports at the crate root) suggests intentional encapsulation — consumers must navigate the module hierarchy explicitly, keeping dependencies visible and reducing coupling between subsystems.

## Overview

`sandbox` is a Rust-based backend server for **ChoirOS** that implements an **actor-based architecture** with a REST and WebSocket API. It serves as a per-user runtime instance, combining an actor system, event sourcing, AI/LLM integration, and an embedded UI into a single deployable binary.

## Key Functionality (Library Modules)

- **`actors`** — Actor definitions and lifecycle management, built on the `ractor` actor framework.
- **`supervisor`** — Actor supervision trees (with an optional `supervision_refactor` feature flag for in-progress restructuring).
- **`api`** — HTTP REST and WebSocket endpoints, powered by `axum` with CORS and static file serving via `tower-http`.
- **`app_state`** — Shared application state wired through the server.
- **`baml_client`** — Auto-generated client for [BAML](https://docs.boundaryml.com/) (Boundary AI Markup Language), providing structured LLM interaction.
- **`tools`** — Tooling integrations, including pseudo-terminal support (`portable-pty`) and external HTTP calls (`reqwest`).
- **`markdown`** — Markdown parsing and processing (`pulldown-cmark`, `regex`).
- **`runtime_env`** — Runtime environment and configuration management (`.env` via `dotenvy`, TOML config parsing).
- **`observability`** — Logging and tracing instrumentation (`tracing` / `tracing-subscriber`).
